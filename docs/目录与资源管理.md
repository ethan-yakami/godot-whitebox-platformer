# 目录与资源管理

本文档用于说明当前 Godot 白盒原型的目录职责，以及后续新增资源时的放置规则。目标是让关卡、脚本、美术、音频、UI、文档和工具各归其位，避免资源引用混乱。

## 当前项目目录

| 路径 | 用途 | 管理建议 |
| --- | --- | --- |
| `project.godot` | Godot 项目配置文件，记录主场景、输入映射、自动加载脚本等设置。 | 优先通过 Godot 编辑器修改，手动编辑前先确认配置含义。 |
| `.godot/` | Godot 编辑器生成的本地缓存和导入状态。 | 不作为正式资源管理目录，不手动整理其中内容。 |
| `scenes/` | 游戏场景文件，例如主流程场景和关卡编辑器场景。 | 放置可直接被 Godot 打开的 `.tscn` 场景；主入口、编辑器、复用场景可以继续放在这里。 |
| `levels/` | 独立关卡场景文件，例如 `level_1.tscn` 到 `level_5.tscn`。 | 只放关卡实例或关卡场景，命名建议使用 `level_数字_说明.tscn` 或延续当前 `level_数字.tscn`。 |
| `scripts/` | 游戏核心 GDScript 逻辑，例如玩家、主流程、全局状态、HUD、输入限制、敌人等。 | 一类系统一个脚本，避免把关卡数据、玩家控制、UI 显示混在同一个文件里。 |
| `scripts/components/` | 白盒组件脚本，例如平台、水域、尖刺、弹跳垫、存档点、祭坛等。 | 放置可复用的玩法组件脚本；新增机关或交互物优先放这里。 |
| `docs/` | 需求、实现方案、资产需求、测试说明、关卡编辑指南等开发文档。 | 所有面向策划、程序、美术协作的说明都放这里；文档命名建议保留编号或清晰主题。 |
| `tools/` | 项目检查、行为验证、静态检查等辅助脚本。 | 只放开发工具，不放游戏运行时脚本。工具脚本应能从项目根目录运行。 |
| `analysis_extracts/` | 从策划文档、PPT、Excel 等资料中抽取出的参考内容和图片。 | 作为参考资料归档，不直接作为最终游戏资源；确认可用后再整理到正式资源目录。 |
| `icon.svg`、`icon.svg.import` | Godot 项目图标及其导入文件。 | 保持在根目录即可；`.import` 文件由 Godot 维护。 |

## 后续建议新增目录

当前项目仍是白盒阶段，正式美术和音频资源不多。等开始接入正式资源时，建议在项目根目录新增 `assets/`，并按资源类型拆分：

| 建议路径 | 用途 | 示例 |
| --- | --- | --- |
| `assets/art/characters/` | 玩家、敌人、Boss、残影等角色图像或动画帧。 | `player_run.png`、`ghost_idle.png` |
| `assets/art/environment/` | 平台、草地、湖泊、洞穴、古塔等环境资源。 | `grass_platform.png`、`tower_wall.png` |
| `assets/art/props/` | 路灯、浆果、石碑、祭坛、机关等道具资源。 | `checkpoint_lamp.png`、`altar.png` |
| `assets/ui/` | HUD、按钮、图标、菜单背景、字体相关资源。 | `heart_icon.png`、`key_limit_panel.png` |
| `assets/audio/bgm/` | 背景音乐。 | `level_1_theme.ogg` |
| `assets/audio/sfx/` | 跳跃、受伤、冲刺、存档、获得传承等音效。 | `jump.wav`、`checkpoint.wav` |
| `assets/vfx/` | 特效贴图、粒子材质、传承获得和受伤反馈资源。 | `legacy_burst.png`、`hit_flash.png` |
| `assets/fonts/` | 游戏字体和 UI 字体。 | `ui_font.ttf` |

新增正式资源后，场景和脚本只引用 `assets/` 中的正式文件；`analysis_extracts/` 继续作为原始资料和参考图归档。

## 命名规则

- 文件名建议使用小写英文、数字和下划线，例如 `player_idle.png`、`level_1_grassland.tscn`。
- Godot 资源路径尽量稳定。已经被场景引用的资源不要随意移动或改名；如需移动，优先在 Godot 编辑器中操作，让引用自动更新。
- 白盒资源可以使用功能名命名，例如 `spike_strip.gd`、`bounce_pad.gd`；正式资源可以增加主题或状态，例如 `spike_cave_idle.png`。
- 关卡文件保留数字顺序，方便主流程按关卡编号加载。
- 文档可以继续使用编号前缀，例如 `01_需求文档.md`，让阅读顺序清楚。

## Godot 导入文件注意事项

- `.import` 文件是 Godot 对图片、音频等资源生成的导入记录，通常应跟随原始资源一起保留。
- 不要只删除 `.import` 文件来尝试清理资源；如果资源不再使用，应先确认场景和脚本没有引用。
- 从外部复制图片、音频到项目后，打开 Godot 编辑器让它完成导入，再检查生成的 `.import` 文件。
- 如果移动正式资源，建议在 Godot 编辑器的 FileSystem 面板中移动，减少引用失效风险。

## 资源进入项目的流程

1. 先把原始策划资料、截图、PPT 导出图放入 `analysis_extracts/`，仅作为参考。
2. 确认某个资源要进入游戏后，按类型整理到 `assets/` 下对应目录。
3. 在 Godot 编辑器中检查导入设置，例如纹理过滤、音频循环、压缩方式。
4. 再把资源挂到场景、组件或 UI 上。
5. 最后运行 `tools/` 中的检查脚本，确认项目结构和行为没有被破坏。

## 不建议随意移动的内容

- `project.godot`
- `scenes/`
- `levels/`
- `scripts/`
- `scripts/components/`
- 已经被 Godot 场景引用的图片、音频、字体和 `.tscn` 文件

这些文件或目录可能被 Godot 的资源路径直接引用。移动前应先确认引用关系，移动后需要打开相关场景检查是否有缺失资源。

## 当前阶段建议

当前仍处于白盒玩法验证阶段，最重要的是保持 `scenes/`、`levels/`、`scripts/` 的结构稳定。正式资产接入前，可以先只维护文档和验证工具；当美术、音频、UI 开始进入项目时，再创建 `assets/` 并按本文档规则归档。
